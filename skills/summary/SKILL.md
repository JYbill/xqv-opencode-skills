# summary

这个 skill 用来把一个代码文件或一个小模块，总结成可维护的技术说明文档。重点不是复述源码，而是帮助人快速理解这段代码负责什么、怎么运行、改它会影响哪里，以及代码里还有哪些待办。

## 适用场景

当用户要求根据某个文件、某个 tool、某个 service、某条接口链路生成总结时，优先使用这个 skill。

当用户明确要求包含核心逻辑、详细代码逻辑、影响范围、TODO 注释、文件名、方法名、行号时，也使用这个 skill。

如果用户只是问一句代码做什么，可以直接简短回答，不需要生成完整总结文档。只有用户要求沉淀文档、写入 docs、生成总结、整理模块说明时，才进入完整流程。

## 非目标

这个 skill 只负责总结和沉淀代码理解，不负责顺手改业务代码，不负责做完整 code review，也不负责生成未来规划或改造方案。

如果阅读过程中发现明显 bug、风险或文档与代码不一致，可以在总结里客观写入当前边界或影响范围，但不要把总结文档写成问题清单、重构计划或审查报告，除非用户明确要求。

## 工作方式

先读取用户指定的主文件。不要只看文件名猜职责，也不要只根据已有记忆写总结。

读完主文件后，继续顺着 import、核心函数调用、schema、type、service、prompt、controller、query 方法查直接依赖。目标是把当前模块的真实执行链路看完整，但不要无限扩散到无关模块。

如果主文件只是入口层，要继续找到真正承载业务逻辑的 logic、service 或 query 文件。总结时需要讲清入口层做什么，核心业务层做什么，不能把入口层误写成全部逻辑。

遇到 schema、类型文件和 prompt 文件时，需要判断它们是否会影响入参、出参、调用方式或展示口径。只要会影响，就应进入影响范围章节。

查 TODO 时要搜索主文件、直接依赖文件，以及当前模块目录下相关文件。搜索关键词至少覆盖 TODO、todo、FIXME、fixme、待办。没有查到就明确写没有发现，不要编造待办。

## 文档结构

生成总结时，首要章节必须是核心逻辑。这个章节要先说明模块定位，再说明主流程，再说明关键设计取舍。写法要像给后续维护者交代上下文，不要像在逐行翻译代码。

详细代码逻辑章节要标注文件名、方法名和行号。这里可以按调用顺序展开，从入口函数到核心编排函数，再到关键算法函数。引用格式优先使用 `path/to/file.ts:line`，如果同一段逻辑跨多行，只标关键起始行即可。

影响范围章节要说明改动当前模块可能影响哪些文件和模块。重点写真实依赖关系，比如 schema、type、prompt、service query、controller、上游调用方、下游聚合逻辑。不要泛泛写“可能影响系统稳定性”这种空话。

TODO 注释章节要把代码中存在的 TODO、FIXME、待办注释归纳出来。每条要说明它在哪个文件、哪一行、表达了什么未完成事项。如果没有 TODO，就写明已检查范围，并说明没有发现。

如果代码里有明显边界、容易误解的字段口径、空结果协议、精度处理、权限约束或时间窗口逻辑，可以补一个当前边界章节。这个章节只写真实代码已经体现的边界，不写未来规划。

## 写作要求

全文保持人类友好。句子要短，逻辑要顺，先讲结论，再讲细节。不要把源码大段贴进文档，除非用户明确要求源码示例。

代码引用要克制。优先引用文件路径、方法名和行号，不要贴大量源码。通常每个关键点引用一处行号就够了。

不要复杂化。不要为了显得完整而扩展无关历史、架构背景、理论解释或不在当前代码中的推测。

不要复读同一信息。核心逻辑讲过模块定位后，详细代码逻辑就只补执行顺序和关键函数，不要再重复同一段介绍。

写影响范围时要具体。应该写清楚某个 schema 会影响结构化输出校验，某个 type 会影响 TypeScript 类型契约，某个 prompt 会影响模型何时调用 tool，某个 query 会影响字段口径。

写 TODO 时要诚实。没有就是没有，不要为了章节完整强行找问题。

## 推荐章节模板

文档可以按下面的顺序组织。

`# 模块总结`

`## 核心逻辑`

说明这个模块负责什么、输入是什么、输出是什么、主流程怎么走、核心设计是什么。

`## 详细代码逻辑`

按入口函数、权限或参数处理、核心编排函数、关键算法函数、输出组装的顺序写。每段都带文件名和行号。

`## 影响范围`

写改动当前模块会影响的 schema、type、prompt、调用方、查询层、业务聚合层和相关模块。

`## TODO 注释归纳`

写检查到的 TODO、FIXME、待办注释。没有时写清检查范围和结果。

`## 当前边界`

可选章节。只有代码里确实存在容易误解的边界时才写。

## 行号要求

行号必须来自实际读取或搜索结果。不要凭印象写行号。

如果文件刚被编辑过，需要重新读取或搜索相关片段，确保行号没有过期。

如果行号密集，不要每一行都标。选关键方法定义、关键分支、关键调用、关键返回结构即可。

## 输出到文件时

总结文档默认存放在 `docs/spec/*` 下对应模块目录。先根据主文件所属业务域和调用链路找到最合适的模块文件夹，比如 MCP 课程相关内容放到 `docs/spec/mcp/course`。如果总结同时涉及多个模块，就需要分别更新这些模块对应目录下的文档，不能只放到其中一个模块里造成上下文缺失。

文档命名要能直接看出主题。默认使用“业务名总结.md”或“业务名工具总结.md”这类中文名称，例如 `课次详情工具总结.md`。如果目标目录里已经存在同主题文档，优先更新已有文档，不要再创建一份内容重叠的新文档。只有主题确实独立，或者用户明确要求新建时，才创建新文件。

如果用户要求写入 docs 或指定路径，先查看目标目录已有文档命名和风格。新文档应尽量沿用相近命名和表达方式，但不要覆盖已有文件，除非用户明确要求覆盖。

写入前要确认目标文件是否已经存在。已存在时先读取内容，再决定是覆盖、追加还是更新。用户没有说覆盖时，优先新建语义清楚的文件。

写入后检查 git diff 或 git status，确认只产生预期文档变更。如果发现工作区还有其他既有改动，只说明它们存在，不要擅自处理。

## 质量标准

一篇合格总结应该让读者不用打开源码，也能知道这个模块的责任、主链路、关键算法、影响文件和 TODO 状态。

一篇合格总结也应该让读者想继续排查时，能根据文档里的文件路径和行号快速跳到源码。

如果总结读起来像源码逐行翻译，说明还不够好。应该重新压缩成模块行为、数据流、关键约束和影响范围。
